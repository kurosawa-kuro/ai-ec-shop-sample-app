# デモ映え機能アイデア — スマホで見せて魅力が伝わる追加機能

2026-07-06 の相談の書き出し。仕様・設計・実装方針は未検討（別フェーズ）。

## 前提

このアプリは売上目的の EC ではなく、**「AI 関連の仕事をしている人が作った、AI 活用 Web アプリ」をスマホで短時間に伝えるためのアピール・コミュニケーション用デモ**。

評価観点は次の 4 つ。

1. スマホで見せたときに短時間で魅力が伝わるか
2. AI 活用が自然に伝わるか
3. 会話のきっかけになるか
4. 「AI の仕事をしている人が作った」と印象に残るか

## 診断：足りないのは機能より「デモ映え」

P0/P1（コンシェルジュ中核・多ターン相談・3品比較・リランキング・添え書き・店舗インサイト）は完了済みで、「AI を業務導線に組み込む」骨格は十分強い。1〜3 分のスマホデモで残るボトルネックは 3 つ。

1. **AI の価値が出るまでに入力が必要** — 相手の前で長文を打つのはデモの死に時間
2. **一番凄い部分（ID 接地・正規化・フォールバック）が画面に見えない** — エンジニアには刺さるが、見た目は「普通に商品が出た」だけ
3. **相手が participant になれない** — 見せるだけだと「ふーん」で終わり、会話が生まれない

追加機能はこの 3 点を潰すものに絞る。ロードマップ P2 の破綻条件（普通の EC 機能を足さない）とも矛盾しない。

## アイデア（優先順）

### 1. 音声入力で相談（Web Speech API）

マイクボタンを押して「妻に5000円くらいで、香りが強すぎないギフト」と**話すと 3 品出てくる**。

- スマホデモとの相性が最強。相手にスマホを渡して「話してみてください」と言える唯一の機能
- ChatGPT 的な「AI と話す」イメージに直結し、AI 活用が一瞬で伝わる
- 実装は軽い部類（Web Speech API）

### 2. AI の思考過程を見せる演出（処理の実況表示）

回答待ちのスピナーを「条件を整理中… → 36 品から候補を絞り込み中… → 選んだ理由を言語化中…」のような**段階ナレーション**に変える。

- 待ち時間が退屈な空白から「AI が働いている見せ場」に変わる
- 実際のパイプライン（接地→絞り込み→正規化）をそのまま演出にするだけなので、話のネタとして嘘がない

### 3. 「AI の裏側」トグル

提案結果の下に「🔍 AI の裏側を見る」を置き、AI に渡したカタログ・返ってきた生 JSON・無効 ID の除外結果などを見せる。

- 非エンジニアには「ちゃんと制御されてる感」、エンジニアには「チャット UI を貼っただけじゃない」差別化が一発で伝わる
- **「AI 関連の仕事をしている人が作った」印象に一番効く**

### 4. ワンタップ・デモシナリオ（30 秒ツアー）

トップに「30 秒で AI 相談を体験」ボタンを置き、定番の相談例が自動入力→提案→比較まで流れる。

- 電波が悪い・時間がない場面でも**デモが必ず成立する保険**
- 既存のフォールバック設計と同じ思想（デモの成功率を機能で担保する）

### 5. 提案結果のシェア（QR / リンク）

提案 3 品と理由を URL に載せて QR 表示し、相手のスマホに持ち帰ってもらう。

- デモがその場で終わらず印象が残る
- localStorage 前提の現構成でも、状態を URL エンコードすれば軽く作れる範囲

## 見送り

| アイデア | 見送り理由 |
|---|---|
| 画像入力（写真から相談） | DeepSeek のビジョン対応が制約。モデル構成を変える覚悟があるときだけ |
| レビュー・お気に入り等の一般 EC 機能 | ロードマップ P2 の破綻条件どおり、コンセプトが薄まる |
| 多言語対応・接客キャラ | デモの尺を食うわりに「AI の仕事をしている」証明にならない |

## 結論

一番のおすすめは **1（音声入力）＋ 2（思考過程の実況）** の組み合わせ。「スマホに話しかける → AI が考えている様子が見える → 理由つきで 3 品出る」という 30 秒の流れができ、4 観点すべてを満たす。3（裏側トグル）は相手がエンジニア寄りのときの二の矢。

## 技術メモ: 音声対応は React でできるのか（2026-07-06 追記）

結論: **できる。Flutter は不要。** 音声入出力はブラウザ標準 API として提供されており、React はそれを呼ぶだけ。フレームワークの制約ではなくブラウザ対応の問題になる。

### 音声入力（聞き取り）— 2つの実装ルート

1. **Web Speech API（SpeechRecognition）**: ブラウザ標準の音声認識で、追加コストゼロ・サーバー不要。マイクに話すとテキストが返るので、既存の相談入力に流し込むだけで成立する（アイデア 1 の想定ルート）。Android Chrome が最も安定、iOS Safari も 14.5 以降で対応（webkit プレフィックス付き）。Firefox は非対応だが、**このアプリは自分のスマホで見せるデモ**なので自分の端末で動けばよく、この弱点はほぼ問題にならない。日本語認識の精度も日常語彙なら実用レベル。
2. **MediaRecorder で録音 → STT API**: マイク音声を録り、Whisper 系や Google Speech-to-Text で文字起こしする方式。全ブラウザで確実に動き精度も高い代わりに、API コストとサーバー側の口が必要。ただし本プロジェクトには既に Supabase Edge Functions（`concierge` / `recommend-products`）で「API キーをブラウザに出さない」ゲートウェイ構造があるため、同じパターンで `transcribe` 関数を 1 本足すだけで既存アーキテクチャに素直に収まる。コストも Whisper 系なら 1 分あたり 1 円未満で、月額基準的に問題にならない規模。

### 音声出力（読み上げ）

`speechSynthesis`（ブラウザ標準 API）が全主要ブラウザで動く。日本語音声の質は端末 OS 依存（iOS はまずまず、Android は機種差あり）。「デモで聞かせて恥ずかしくない声」にしたければ TTS API の選択肢もあるが、まず標準 API で試すのが順当。

### Flutter が必要になる条件

常時待ち受け（ウェイクワード）、バックグラウンド録音、オフライン認識などネイティブ機能が要るときだけ。「画面を開いてマイクボタンを押して話す」という今回のデモ用途は Web の守備範囲に完全に収まる。

### 推奨方針

**まず Web Speech API で作って自分の端末で精度を確かめ、不満が出たら Edge Function 経由の STT API に差し替える**二段構え。差し替えても「マイクボタン → テキストが入る」という UI 側の作りは変わらないため、最初の選択で手戻りは発生しない。
