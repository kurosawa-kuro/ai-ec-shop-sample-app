# フロントエンドデザインシステム

## 採用方針

Lumière Select の UI 基盤は **Pico CSS + Lumière UI foundation** とする。

`/home/ubuntu/repos/portfolio/admin-pico` から、次の設計原則を採用した。

- theme / layout / components / page styles の責務を分ける
- アプリシェルとナビゲーションをページから分離する
- 色、余白、角丸、コントロール高さ、フォーカスリングをトークンで管理する
- モバイルはリンクを隠すのではなく、サイドドロワーへ集約する
- ページ固有の見せ場以外は共通コンポーネントを使う

Tailwind 版 React Admin は採用しない。管理画面としての部品数と完成度は高いが、Lumière Select
は顧客向け EC が中心であり、全面移植するとユーティリティクラス、依存パッケージ、既存 Pico
との競合が増える。既存の API 呼び出し・状態管理・購入フローを維持する目的に対して変更面積が過大なためである。

## CSS の責務

| ファイル | 責務 |
|---|---|
| `app/src/styles/theme.css` | 色、余白、角丸、共通寸法、フォーカスなどのデザイントークン |
| `app/src/styles/design-system.css` | AppShell、ヘッダー、モバイルドロワー、共通ページ見出し、パネル、空状態、フォーム寸法 |
| `app/src/index.css` | 商品カード、AI 推薦、購入フロー、店舗インサイトなどページ固有の表現 |

読み込み順は Pico → theme → page styles → design-system。design-system を最後に置き、共通契約がページ固有 CSS
に上書きされないようにする。

## 共通コンポーネント

- `AppShell`: スキップリンク、ヘッダー、本文、フッター、カートアニメーションを統括する。
- `Header`: デスクトップナビゲーションとモバイルサイドドロワーを同じリンク定義から生成する。
- `PageHeader`: eyebrow、アイコン、`h1`、リード文、補助アクションの並びを統一する。
- `Panel`: `section` / `aside` などの意味論を保ったまま、共通 surface を適用する。
- `EmptyState`: 空状態の見出し、説明、アクションの配置を統一する。
- `CheckoutSteps`: カートから注文完了までのステップ状態を統一する。

## 実装ルール

1. 新しいページは `AppShell` 配下に置き、本文コンテナには `site-page` を使う。
2. 通常ページの先頭は `PageHeader` を使う。トップのヒーロー、注文完了など明確な理由がある画面だけ独自構造を許可する。
3. 枠線付き surface は `Panel`、データなし表示は `EmptyState` を使う。同用途の新規カードをページ内に再実装しない。
4. 色、余白、角丸、コントロール高さを直値で増やさず、まず `theme.css` のトークンを使う。
5. `!important` は通常レイアウトに使わない。現在の例外は `prefers-reduced-motion` を確実に優先するアクセシビリティ規則のみ。
6. inline style は座標や CSS custom property など実行時にしか決まらない値に限定する。
7. モバイルで導線を非表示にしない。収まらないナビゲーションはドロワーへ移す。
8. タップ対象は 44px 以上、入力欄は共通高さ、キーボードフォーカスは `--focus-ring` を使う。
9. 新しいブレークポイントをページごとに増やさず、基本は 760px（ナビゲーション）と既存の 920px / 640px（ページグリッド）に合わせる。

## 品質ゲート

UI 変更後は次を実行する。

```bash
cd app
npm run build
npm run lint
npx playwright test
```

E2E は既存業務フローに加え、390px / 820px / 1440px で主要ページに横スクロールが発生しないこと、モバイルドロワーから主要導線へ移動できること、ブラウザのダーク設定でもライトテーマを維持することを検証する。
